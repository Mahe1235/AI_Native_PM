Concept 01

The Vibe Check Trap — Why Your AI Output Is Worse Than You Think

You've been reading Claude's output, thinking "yeah, that looks good," and shipping it. That instinct is costing you more than you realize.

Most PMs evaluate AI output the same way: read it once, check if it feels right, maybe fix a sentence, ship it. This is the vibe check — and it is the single most common reason AI skills stay mediocre instead of becoming genuinely reliable.

The vibe check feels efficient. You're a smart PM. You can tell good output from bad output. Except you can't — at least not consistently, and not at scale. Here's why:

  • You're unconsciously editing. When you read Claude's output, your brain automatically fills gaps, smooths transitions, and overlooks missing sections. You're not evaluating the raw output — you're evaluating the output plus your mental corrections. This means you think the skill is better than it actually is.
  • You don't remember failures. Last Tuesday's output was great. Friday's had a weird hallucinated metric. Monday's was fine. You remember the good ones. Cognitive bias gives you an inflated quality assessment.
  • One input hides structural problems. You keep testing with the same type of input — your weekly sprint data, your usual stakeholder format. The skill might fail catastrophically on edge cases you've never tried: a sprint with no completed stories, a stakeholder who needs technical depth instead of executive summary.

The Real Numbers

When PMs run structured evaluations on skills they thought were "working great," they typically find 40–60% pass rates. Not 90%. Not 80%. Roughly half the outputs have issues the PM had been unconsciously fixing every time. The vibe check didn't catch those problems — it hid them.

This isn't a hypothetical. The pattern shows up everywhere AI output meets informal evaluation:

of gen AI users encountered incorrect answers in a 2025 Deloitte survey
CNET

Apple

The Air Canada case is especially instructive — the company argued in court that the chatbot was "a separate legal entity" responsible for its own accuracy. The tribunal rejected this. You are responsible for your AI's output.

Counterintuitive

The vibe check feels like the fastest path. It's actually the most expensive one. Every hour you skip on structured evaluation, you spend two hours later — fixing errors in shipped work, rebuilding stakeholder trust, or manually correcting output that a better prompt would have gotten right automatically. Systematic evaluation isn't extra work. It's the work that reduces all your other work.

Concept 02

Eval-Driven Development — Evals Are How You Build, Not How You Check

The biggest mistake isn't skipping evals. It's treating them as a final step instead of the primary development method.

Here's how most PMs develop an AI skill: write a system prompt, run it once, tweak the prompt, run it again, tweak some more, decide it's good enough, ship it. Evaluation — if it happens at all — comes at the end.

Flip that. Evaluation should consume 60–80% of your AI development time. Not the prompt tweaking. Not the system prompt architecture. The systematic review of what's working and what isn't.

This is eval-driven development, and it works the same way test-driven development works in software engineering: write the test first, then build until it passes.

The Reactive Loop vs. The Eval Loop

The Reactive Loop
How most PMs work
Write prompt → run it → read output → "looks fine" → ship → discover problem in production → panic-fix the prompt → repeat. You're always one step behind your problems. Each fix might create a new issue somewhere else, but you won't know until it surfaces.
The Eval Loop
Eval-driven development
Define what "good" looks like → build test dataset → run skill on all inputs → label pass/fail → identify top failure → fix it → re-run all inputs → measure improvement → repeat until quality target is met. You see problems before they reach anyone else.

Anthropic published a guide on evaluating AI agents that crystallized this insight: teams without evals get stuck in reactive loops — catching issues only in production, where fixing one failure creates others. Teams with evals can upgrade models, change prompts, and ship improvements with confidence because they have measurable baselines.

What an Eval Actually Is

Strip away the jargon and an eval is just a software test for AI output. Instead of testing "does this function return 42?", you're testing "did Claude write a PRD that actually covers the right things?" The components are simple:

  • Input: A specific task you give to your skill (e.g., "Write a stakeholder update for this sprint data")
  • Output: What the skill produces
  • Grading logic: How you decide if the output passes or fails

That's it. You already do this instinctively when you read Claude's output and think "that's not quite right." The difference is doing it systematically — with multiple inputs, clear criteria, and a record of the results.

Counterintuitive

Evals get harder to build the longer you wait. Early on, your product requirements naturally translate into test cases. "The PRD must include a metrics section" is both a requirement and an eval criterion. Wait until you've shipped and now you're reverse-engineering what "good" means from production failures. Start with evals from day one, even rough ones.

Counterintuitive

20 test cases beats 200 when you start. Anthropic recommends 20–50 simple tasks drawn from real failures. Early changes have large effect sizes — a prompt improvement might jump your pass rate from 40% to 70%. You don't need statistical power for that. You need enough variety to surface different failure modes. Start with 15, expand as your skill matures.

Concept 03

Writing Pass/Fail Rubrics — The Skill Behind the Skill

Defining "good" in 2–3 sentences is the hardest and most important part of evaluation. Get this wrong and everything downstream is noise.

Before you can evaluate anything, you need to answer the question: what does "pass" mean? This sounds simple. It's deceptively hard — and it's where most evaluation efforts go wrong.

Why Binary, Not Scales

Use pass/fail. Not 1–5 scales. Not "good/okay/bad." Binary.

Likert scales feel more sophisticated, but they introduce two problems. First, your 3 and my 3 aren't the same 3 — even when you're the only rater, your standards drift between sessions. Tuesday at 9am you're strict; Friday at 4pm you're generous. Second, scales obscure the signal. If an output scores 3/5, what do you do with that? Is it shippable? The binary question — "would I send this to a stakeholder as-is?" — forces clarity.

Define "pass" in 2–3 sentences. No more. If your rubric needs a paragraph, it's measuring too many things at once — split it into separate criteria.

The PM Skill Transfer

Here's the good news: you already know how to do this. A pass/fail rubric is an acceptance criterion. Every PM who's written a user story has written something like "the dashboard must load in under 3 seconds and display all active campaigns." That's a rubric — clear, binary, testable.

Apply the same discipline to your AI output:

Vague Rubric
"Output should be good and comprehensive"
What does "good" mean? What counts as "comprehensive"? Two people reading this rubric will label the same output differently. Your own labels will be inconsistent across sessions. Results: meaningless.
Specific Rubric
"Output must include: (1) the user problem in the first paragraph, (2) a proposed solution with at least one alternative considered, and (3) at least one measurable success metric. Fail if any of the three is missing or fabricated."
Three checkable conditions. Binary pass/fail. Two people would label the same output the same way. Your labels stay consistent across sessions. Results: actionable.

The Rubric Design Checklist

Every good rubric passes four tests:

  • Is each condition independently verifiable? Can you point to a specific part of the output and say "this condition is met" or "this condition is missing"? If a condition requires subjective interpretation ("is the tone appropriate?"), make it concrete ("uses no more than 3 technical terms without definitions").
  • Is failure unambiguous? If someone fails an output, could the writer of the rubric tell which condition wasn't met? If the rubric is "must be accurate and well-structured," and an evaluator marks "fail," you don't know if accuracy or structure was the issue. Split them.
  • Does it match the actual use case? A rubric for an executive summary should care about brevity and top-line clarity. A rubric for a technical spec should care about edge case coverage. The same skill used for different audiences might need different rubrics.
  • Can you apply it in under 30 seconds per output? If evaluation takes five minutes per output, you won't do it consistently. Good rubrics are fast to apply — scan for the required elements, check, move on.
Counterintuitive

Separate rubrics per dimension beat one combined rubric. Research on LLM evaluation consistently finds that evaluating one criterion at a time produces more reliable results than evaluating multiple criteria simultaneously. "Is this factually accurate?" and "Is this well-structured?" are different questions. Rate them separately. When you later identify that 80% of failures are accuracy-related and only 10% are structure-related, you'll know exactly where to invest your prompt improvement effort.

Practical Tip

Write your rubric before you look at any outputs. If you write it after, you'll unconsciously tailor it to match what your skill already produces — which defeats the purpose. Think of it as writing the acceptance criteria before the sprint starts, not after the feature ships.

Concept 04

Building Your Test Dataset — 15 Inputs That Reveal the Truth

Your eval is only as good as the inputs you test with. Here's how to design a test dataset that surfaces real failures instead of confirming your assumptions.

You need at least 15 test inputs for your first eval. Not 3 (too few to see patterns), not 50 (overkill for initial evaluation — save that for Module 06). Fifteen inputs give you enough variety to surface different failure modes while staying manageable enough to label by hand in under an hour.

The critical mistake: don't just use 15 copies of your typical input. If all 15 are clean, well-structured sprint data from your best-performing team, your eval will tell you the skill works for easy cases. You already knew that. You need inputs that stress-test the edges.

Dimension-Based Synthetic Generation

When you don't have 15 real test cases ready, generate them systematically by combining dimensions. For a PM stakeholder update skill, the dimensions might be:

DimensionVariation AVariation BVariation C
ComplexitySimple sprint (5 stories, all green)Medium sprint (12 stories, 2 blockers)Messy sprint (20 stories, scope change, team conflict)
AudienceVP of Engineering (wants technical detail)CEO (wants business impact only)Cross-functional stakeholders (mixed technical fluency)
Data qualityClean, complete JIRA exportMissing velocity data, some stories unlabeledPartially in a different format (migrated from Linear mid-sprint)

Three dimensions × three variations = 27 possible combinations. Pick 15 that cover the most important intersections, prioritizing combinations your real work actually encounters. One high-complexity test case with missing data for a CEO audience reveals more than five easy cases ever will.

Real Data vs. Synthetic Data

Use real data whenever possible. Your actual sprint exports, real stakeholder emails, genuine feedback tickets. Real data has the messiness, inconsistency, and edge cases that synthetic data often misses. Save synthetic generation for combinations your real data doesn't cover — the messy sprint you haven't had yet, the stakeholder audience you haven't written for.

Counterintuitive

Your "hardest" test cases produce the most valuable learning. PMs instinctively build test datasets that look like their typical work — clean inputs, clear requirements, standard audiences. But the messy cases — the sprint with a scope change mid-week, the stakeholder who needs both technical and business framing, the feedback dataset with contradictory user quotes — are where you learn what your skill actually can't do. Build your dataset to be 40% typical, 40% edge cases, 20% deliberately adversarial.

Practical Tip

Keep a running "failure collection." Every time your skill produces something you manually fix, save the input that caused it. Over weeks, you'll build a test dataset sourced from real failures — far more valuable than any synthetically generated set. By Module 06, this collection becomes one of your most important assets.

Concept 05

The 4-Step Error Analysis Protocol

A systematic method borrowed from qualitative research that transforms "this output seems off" into "here is the exact failure pattern and here is how to fix it."

This is the core of the module — the protocol you'll use for every eval you run from now on. It borrows from qualitative research methods (open coding and axial coding) and applies them to AI output evaluation. You don't need to know the academic theory. You just need the four steps.

01
Build Test Dataset
15 inputs covering typical cases, edge cases, and adversarial cases. Run all through your skill.
02
Open Coding
Read each output. For every failure, journal the first thing you notice is wrong. Don't fix — just note.
03
Axial Coding
Group your failure notes into categories. Name each category. Count how many outputs hit each one.
04
Fix & Re-eval
Fix the #1 failure category only. Re-run all 15 inputs. Measure: did pass rate improve? Repeat.

Step 1: Build Your Test Dataset

You learned how in Concept 04. Gather your 15 inputs, run all of them through your skill, and save every output. Don't read them yet — just collect them. You want the full set before you start evaluating, so your standards don't drift mid-assessment.

Step 2: Open Coding

Now read each output against your rubric. For every failure, write down the first thing you notice that's wrong. Not everything that's wrong — just the first issue that jumps out. This matters because early failures tend to be the most salient and systematic ones.

Don't agonize over whether your notes are perfect. You're building a dataset of observations, not writing a paper. Speed matters — you want to get through all 15 outputs while your judgment is fresh.

Step 3: Axial Coding

Now look across all your failure notes. You'll see patterns. Group the similar failures together and give each group a name. This is your failure taxonomy — the categories of ways your skill breaks. Concept 06 goes deeper on taxonomies, but here's the immediate step: count how many outputs fall into each category.

Step 4: Fix and Re-Eval

Fix the #1 category only. The one with the most failures. Make a single, targeted change to your system prompt — add a constraint, add an example, rewrite a section. Then re-run all 15 inputs and re-label everything.

Did the pass rate go up? Did the #1 category shrink? Did any new failure mode appear? Record all of this. You're not done — you repeat this cycle until you either hit your quality target or reach a point of diminishing returns (usually 3–5 cycles).

Core Principle

Error analysis should consume 60–80% of your AI development time. Most people spend 90% tweaking prompts and 10% checking if the changes helped. Flip that ratio. The 4-step protocol tells you exactly where to invest your prompt-engineering effort — you're never guessing about what to fix next.

Concept 06

Failure Taxonomies — Naming What's Going Wrong

A named failure is half-fixed. An unnamed failure recurs forever. Your taxonomy is the map that turns "this output seems off" into "here is the pattern and here is the fix."

When you finish axial coding (Step 3 above), you have a list of failure categories. That list is your failure taxonomy — the single most valuable artifact your eval produces. More valuable than the pass rate number. More valuable than any individual fixed output. The taxonomy tells you how your skill fails, which tells you exactly what to fix.

A Real Taxonomy Example

Here's what a failure taxonomy actually looks like for a PM stakeholder update skill, after evaluating 15 outputs:

Failure Taxonomy — Stakeholder Update Skill v1

Missing risks section

Wrong audience tone

This taxonomy tells you everything: the #1 problem is hallucinated data (invented metrics, made-up dates). That's where 40% of your failures live. Fix this one category — add an explicit constraint like "never invent metrics; if data is missing, state what's missing" — and your pass rate might jump from 53% to 75% in a single prompt change.

Counterintuitive

The most common failure mode is almost never what you expect. Before running evals, most PMs predict their skill's #1 problem will be "not enough detail" or "wrong format." The actual #1 failure is almost always something they didn't anticipate — hallucinated data, skipped sections, wrong audience framing, or invented KPIs. You genuinely cannot predict failure modes from introspection. You have to measure.

Taxonomy Naming Discipline

Name your categories precisely. "Bad output" is useless. "Hallucinated data" is actionable. "Tone issues" is vague. "Uses technical jargon for non-technical audience" is fixable. The quality of your category names determines the quality of your fixes. Each name should immediately suggest a prompt change.

When to Split Your Skill

If your taxonomy shows 3+ failure modes that can't be fixed without contradicting each other — for example, "needs more technical depth" AND "too much jargon for non-technical readers" — the skill is trying to do too much. Split it into two skills with different audience-specific system prompts. A "Technical Stakeholder Update" and an "Executive Stakeholder Update" will each perform better than one skill trying to serve both audiences.

Concept 07

The Fix-One-Thing Cycle — Why Fixing Two Things at Once Is Worse

Disciplined single-variable iteration sounds slow. It's the fastest path to a reliable skill.

You've identified your #1 failure mode. You also see #2 and #3 in the taxonomy. The temptation is irresistible: fix all three at once, re-run, save time.

Don't.

When you change two things simultaneously and the pass rate goes from 53% to 67%, you don't know which change helped. Maybe the first fix improved things by 20 points and the second made things worse by 6 points, and you're seeing the net. You'll carry that second, harmful change forward forever.

The fix-one-thing-and-re-eval cycle works like this:

  1. Fix only the #1 failure category. Make one change to your system prompt — add a constraint, rewrite an instruction, add an example that demonstrates the correct behavior.
  2. Re-run all 15 inputs. Not just the ones that previously failed. Sometimes fixing one thing breaks something that was passing before. You need the full picture.
  3. Re-label everything. Did the #1 category shrink? Did any new category appear? What's the new pass rate?
  4. Record the change. "v2: Added 'never invent metrics' constraint. Pass rate: 53% → 73%. Hallucinated data: 6 → 1. New issue: 2 outputs now missing velocity context (was available in data but model skipped it)."
  5. Repeat. Now fix the new #1 category (which might be the old #2, or might be the new issue you just discovered).

Three to five cycles typically gets a skill from initial quality to reliably good. Track each cycle in a simple table — this becomes your prompt changelog, and it's the first thing you'll reference when the skill needs updating later.

Counterintuitive

Your skill will sometimes get worse before it gets better. You fix hallucinated data by adding a strict "only cite numbers from the input" constraint. Now the skill stops inventing numbers — but it also stops calculating totals that the source data clearly supports. This is a new failure mode caused by an over-correction. The fix-one-thing cycle catches this immediately because you re-run all 15 inputs after every change. The fix-three-things approach would bury it in noise.

Practical Tip

Version your system prompts explicitly. "Stakeholder Update Skill v1" → "v2" → "v3." Write a one-line changelog entry for each version: what you changed and why. Module 06 formalizes this into prompt versioning, but start the habit now. A month from now, you'll be glad you can trace why your skill behaves the way it does.

Concept 08

Grader Types — You vs. Code vs. LLM-as-Judge

There are three ways to evaluate AI output. Each has a sweet spot. PMs who use the right grader for each criterion save hours without sacrificing quality.

So far, you've been the grader — reading each output against your rubric and labeling pass/fail. That works for 15 inputs. It doesn't work for 150. And it definitely doesn't work when you want to run evals weekly as part of a feedback loop (Module 06).

There are three types of graders, each suited to different evaluation criteria:

Human Grading
You Read It
You read the output, compare it to your rubric, label pass/fail. The gold standard — most accurate, most nuanced. But slowest and most expensive. Standards drift between sessions.
Best for: Initial eval (first 15), calibrating other graders, subjective quality dimensions ("would I actually send this?"), final sign-off before shipping
Code-Based Grading
String Matching
Simple programmatic checks — does the output contain a "Risks" section? Is it under 500 words? Does it include at least one number? Fastest and most reliable. But can only check structural features, not quality.
Best for: Structural checks (section presence, word count, format compliance), regression tests after prompt changes, checks you'll run hundreds of times
LLM-as-Judge
Claude Grades Claude
A second Claude instance evaluates the first's output using a rubric you provide. Scales like code, handles nuance like humans. But has systematic biases and requires calibration.
Best for: Quality dimensions at scale (clarity, accuracy, audience-appropriateness), running evals on 50+ outputs, continuous monitoring in production

The PM Grading Stack

In practice, you'll use a combination. For a stakeholder update skill, the grading stack might look like:

CriterionGraderWhy this grader
Has "Progress," "Risks," and "Asks" sectionsCodeSimple string check — fast, deterministic, never wrong
Under 400 wordsCodeWord count — code is perfect for this
No jargon for non-technical audienceLLM-as-JudgeRequires understanding of context — code can't detect jargon reliably
Risk mitigations are specific, not genericLLM-as-Judge"Specific" is a quality judgment — needs language understanding

Start with human grading for your first eval. It's the foundation — you need to understand your skill's failure modes before you can automate grading. Then layer in code graders for structural checks and LLM-as-Judge for quality checks. Keep human grading for the final 10% spot-check that keeps everything calibrated.

Counterintuitive

The most capable grader isn't always the best grader. Using LLM-as-Judge for "does the output have a Risks section?" is like hiring a senior engineer to count lines of code. A simple string match is faster, cheaper, more reliable, and never hallucinates. Match the grader to the complexity of the criterion. Reserve LLM judges for criteria that genuinely require language understanding.

Concept 09

LLM-as-Judge for PMs — Scaling Evaluation Without Losing Quality

Using Claude to evaluate Claude's output sounds circular. Done right, it's the most practical evaluation technique available — with real limitations you need to understand.

LLM-as-Judge is the technique that lets you run evaluations at scale. Instead of you reading 50 outputs, you write a rubric prompt and have a separate Claude instance grade each output programmatically. Research shows strong LLM judges agree with human evaluators about 80–85% of the time — which is comparable to the agreement rate between two human raters (approximately 81%).

That statistic should simultaneously reassure you and keep you honest: LLM judges are roughly as consistent as humans, which means they're good enough for most evaluation purposes. But "roughly as consistent as humans" also means they make mistakes at a similar rate. This isn't a replacement for your judgment. It's a scalable complement to it.

Writing a Judge Prompt

A judge prompt is a specialized system prompt — except instead of generating content, it evaluates content. The structure follows the same contract pattern you learned in Module 03:

LLM-as-Judge Prompt — Stakeholder Update QualityYou are an evaluation judge. You will be given an AI-generated stakeholder update and the source sprint data it was based on. Evaluate ONLY this criterion: factual accuracy. A PASS means: - Every number, date, and metric in the output appears in the source data - No metrics are invented or extrapolated beyond what the data supports - If data is missing, the output explicitly flags it rather than guessing A FAIL means: - Any number, date, or metric appears that is not in the source data - Any calculation is performed that the source data doesn't support - Missing data is silently filled in rather than flagged Respond with exactly one word: PASS or FAIL. Then on a new line, provide a one-sentence explanation.

Notice what this prompt does: it evaluates one criterion (factual accuracy), defines pass and fail concretely, and asks for a binary verdict. This is the same rubric design discipline from Concept 03, applied to your judge.

Known Biases to Watch For

LLM judges have systematic biases that you need to account for:

BiasWhat happensHow to mitigate
Verbosity biasLLM judges tend to rate longer, more detailed outputs higher — even when brevity was the success criterion. Estimated ~15% score inflation for verbose outputs.Explicitly instruct the judge to penalize verbosity: "Longer does not mean better. Apply the word count criterion strictly."
Position biasWhen comparing two outputs, the judge tends to favor the one presented first (or last, depending on the model). One study found ~40% inconsistency when output positions were swapped.If doing pairwise comparisons, evaluate both orderings (A vs B and B vs A) and only count consistent verdicts.
Self-enhancement biasClaude rates Claude's output slightly higher than it would rate equivalent output from another model. The judge subtly recognizes its own patterns.Consider using a different model as judge than the one generating output. Or calibrate: compare judge verdicts against your human labels on a test set.
SycophancyThe judge may be reluctant to label output as "fail," especially when the output is coherent and well-written. Confident, fluent text gets the benefit of the doubt.Use explicit failure criteria, not just success criteria. Tell the judge what specifically constitutes a fail, not just what constitutes a pass.

The Calibration Protocol

Before trusting your LLM judge, calibrate it against your human labels:

  1. Take your 15 human-labeled outputs from Step 2. You already have verdicts for these.
  2. Run the same 15 through your LLM judge. Compare judge labels to your labels.
  3. Calculate agreement rate. If the judge agrees with you on 12/15 (80%), it's well-calibrated. If it agrees on 8/15 (53%), your judge prompt needs work.
  4. Examine every disagreement. When the judge says PASS and you said FAIL (or vice versa), read the output again. Sometimes the judge caught something you missed. Sometimes the judge is wrong. Adjust the judge prompt based on the pattern of disagreements.

Once calibrated, you can use the LLM judge for larger-scale evals (50+ outputs) and keep human grading for the 10% spot-check that ensures the judge hasn't drifted.

Counterintuitive

LLM judges agree with humans more reliably than humans agree with each other. The ~80–85% agreement rate between LLM judges and human evaluators is comparable to — and sometimes exceeds — inter-annotator agreement between two human raters. Humans are inconsistent too. The advantage of the LLM judge isn't that it's perfect. It's that it's consistently imperfect in predictable, correctable ways.

What Doesn't Work

Buying eval tools before doing it manually. If you haven't manually reviewed 15+ outputs and built a failure taxonomy, you don't know what to measure. Tools like DeepEval, Braintrust, and Langfuse exist and are genuinely useful — but for later, once you know your failure modes and have a calibrated rubric. Manual first. Tooling later. The tools accelerate a process you already understand; they can't replace understanding you don't have yet.

Concept 10

Prompt Caching for Eval Efficiency

Running 15+ inputs through a skill with a long system prompt can get slow and expensive. Prompt caching solves this — your system prompt is processed once and reused.

When you run an eval, you're sending the same system prompt to Claude 15+ times — once for each test input. Without caching, Claude processes that entire system prompt from scratch every time. If your system prompt is 300 words plus uploaded reference documents, that's a lot of repeated work.

Prompt caching fixes this. When enabled, Claude processes your system prompt once, caches the result, and reuses it for subsequent requests. The first call takes the same time as always. Every subsequent call in the eval sprint is faster and cheaper — typically 90% lower latency and significantly reduced cost on the cached portion.

When This Matters for PMs

For a single 15-input eval, the savings are nice but not transformative. Where caching becomes essential is when you're doing the fix-one-thing cycle: you run 15 inputs, make a change, run 15 again, make another change, run 15 again. Three cycles × 15 inputs = 45 API calls with the same system prompt. Caching means that system prompt is processed once, not 45 times.

This matters even more in Module 06, where you'll run eval sprints regularly — monthly checks on your skill's performance against a growing test dataset. Caching makes the difference between "I'll run evals when I have time" and "evals run automatically as part of my workflow."

Practical Tip

If you're running evals through the Claude API (which Module 05 introduces), prompt caching is enabled by marking your system prompt as cacheable. If you're running evals manually in Claude Projects, the conversation-level caching handles this automatically — your system prompt and uploaded docs are processed once per conversation and reused across messages. The key behavior to know: keep your eval in one conversation. Starting a new conversation resets the cache.

Concept 11

From Eval to Improvement — The Bridge to Module 06

Your eval isn't a one-time event. It's the foundation of a system that makes your AI skills genuinely better over time — if you design the feedback loop right.

By now you've run your first structured eval. You have a pass rate, a failure taxonomy, and a prompt that's measurably better than where you started. But this raises the question: when do you eval again?

Your Eval Dataset Is a Personal Asset

Every test case from real work — every input that caused a failure, every edge case that surprised you, every golden example that passed — is a permanent asset. It's your regression test suite. Next time you update your system prompt (for any reason), you re-run the test dataset and check that nothing broke.

This is one of the most underappreciated truths in AI development: your eval dataset compounds in value. Three months from now, you'll have 40+ test cases sourced from real PM work. That dataset represents hundreds of hours of implicit knowledge about what "good" looks like for your specific workflows. No one else has it. It's your competitive advantage as an AI-native PM.

Three Signals It's Time to Re-Evaluate

  • You updated your system prompt. Any prompt change — even a small one — can introduce regressions. Re-run your test dataset. This takes 10 minutes if you've kept your eval organized.
  • The model updated. Claude improves regularly. Sometimes improvements change behavior in ways that interact with your prompt. Teams with eval datasets can upgrade models in days; teams without them face weeks of manual testing.
  • You manually fixed an output. Every time you edit Claude's output before sending it, that's an eval signal. The input that required editing should go into your test dataset. The edit itself tells you what the failure was.

What Module 06 Builds On This

This module taught you how to eval. Module 06 teaches you how to make evaluation a sustainable system:

  • Prompt versioning — tracking changes with changelogs and measured pass rates per version
  • When to split a skill — using your taxonomy to decide when one skill is doing too much
  • Automated feedback capture — using hooks to log every output for review, turning your daily work into continuous eval data
  • Eval sprints — monthly structured evaluations with prompt caching, measuring drift and improvement over time

The key mental shift: evaluation isn't a phase. It's a mode of working. Every time you use an AI skill and notice something about the output — good or bad — that's an evaluation data point. Module 06 teaches you to capture those data points systematically instead of letting them evaporate.

Counterintuitive

Evals that "pass" your skill today can become useless within months. As models improve, evals that were challenging become saturated — the model passes everything. Anthropic observed that frontier models went from 40% to over 80% on SWE-bench Verified in just one year. If your eval pass rate hits 95%+, it's time to raise the bar: add harder test cases, tighten your rubric criteria, or add new dimensions to evaluate. A perfect score means your eval is too easy, not that your skill is perfect.

The PM Superpower

A PM who can run structured error analysis on AI output is extraordinarily valuable. Not just for your own skills — for your product team. When your product ships an AI feature and something "feels off," you're the person who can tell the designer and engineer exactly how the AI is failing: "42% of outputs hallucinate completion dates, 23% skip the risk section, and the rest are fine." That's a specific bug report, not a vague complaint. That's the language engineering teams respond to.

Module 04 Project

The Full Error Analysis Sprint

This project turns everything you learned into a complete evaluation of your best Module 03 skill. By the end, you'll have measurable proof of improvement — and a repeatable process you can apply to any AI skill for the rest of your career.

  1. Write your rubric before looking at outputs. Take your best-performing Claude Project from Module 03. Define pass/fail in 2–3 sentences per criterion. Include at least two criteria: one structural (section presence, word count) and one quality-related (accuracy, audience-appropriateness). Save this rubric — you'll need it for comparison with your LLM judge.
  2. Build your test dataset. Gather 15 inputs using dimension-based generation. At least 5 should be edge cases or adversarial inputs that your skill hasn't seen before. Mix real data (from your actual work) with synthetic data (generated to cover specific dimension combinations). Document each input's source and which dimensions it tests.
  3. Run the 4-step error analysis. Run all 15 inputs through your skill. Label every output pass/fail against your rubric (open coding). Journal the first failure you notice for each failing output. Group failures into categories (axial coding). Count the frequency of each category. You now have your failure taxonomy.
  4. Run the fix-one-thing cycle. Fix only the #1 failure category with a single prompt change. Re-run all 15 inputs. Re-label. Record: what you changed, pass rate before and after, what happened to each failure category. Run at least 2 cycles.
  5. Write an LLM-as-Judge prompt and calibrate it. Write a judge prompt for your top quality criterion. Run it on the same 15 outputs you've already labeled by hand. Compare: where does the judge agree with you? Where does it disagree? Adjust the judge prompt until agreement exceeds 80%. Document the calibration gaps you found.
  6. Compile your eval deliverable. Your final output is an eval spreadsheet (15 rows × columns for input, output, human label, judge label, failure category, version) plus a one-page eval summary: initial pass rate, final pass rate, failure taxonomy with frequencies, prompt changelog, and LLM judge calibration results. This deliverable becomes the input for Module 05's agent building — your evaluated, improved skill is now reliable enough to become a component in a multi-step workflow.
