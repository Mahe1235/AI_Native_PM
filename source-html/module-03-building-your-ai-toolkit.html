Concept 01

Designing AI Workspaces, Not Writing Prompts

Your prompt is roughly 10% of what determines output quality. The other 90% is everything else the model sees. Designing that environment is the real skill.

Most people think the key to getting good AI output is writing a better prompt. That's like thinking the key to onboarding a new hire is writing a better first email. The email matters — but the onboarding package, the documentation, the tools, the team context, and the access permissions matter far more.

The industry has a name for this shift: context engineering. If prompt engineering is about what you say to the model, context engineering is about what the model sees when you say it. Andrej Karpathy framed it well: the LLM is a CPU, the context window is RAM, and your job is to be the operating system — loading exactly the right information for each task.

For PMs, this reframe is liberating. You're not trying to find magic words. You're designing an environment — one where a very capable, very literal new team member has everything they need to do great work.

What Actually Fills the Context Window

Every time Claude generates a response, it sees all of this — not just your message:

What Claude Sees When You Send a Message
~30%
System Prompt / Project Instructions
Your role definition, constraints, output format, examples. The "job description" for this conversation.
~25%
Uploaded Documents & Knowledge
Files in your Project, reference docs, data. The "employee handbook" and relevant briefs.
~20%
MCP Tool Outputs
Live data from JIRA, Slack, Google Drive, Notion — pulled in real-time. The "current state of the world."
~15%
Conversation History
Everything said so far in this chat. The "shared context" from your working session.
~10%
Your Actual Message
The question or instruction you just typed. What most people think of as "the prompt."

The percentages vary, but the principle is consistent: your typed message is a small fraction of what shapes the output. The system prompt, the uploaded docs, the connected tools, the conversation history — that's where output quality is really determined.

Counterintuitive

Research suggests LLM reasoning performance starts degrading around 3,000 tokens of prompt — well below the technical maximum. The practical sweet spot for most task instructions is 150–300 words. More context doesn't mean better output, even with a 200K token window. You want minimum viable context: the smallest, most relevant set of high-signal information. Think of it as designing an onboarding doc, not writing an encyclopedia.

The Workspace Design Mindset

From this point forward, every time you set up an AI interaction, think in terms of workspace design:

  • What role does this specialist need to play? (→ system prompt)
  • What reference materials do they need on their desk? (→ uploaded docs)
  • What live information do they need to pull? (→ MCP connectors)
  • What does great output look like for this task? (→ examples)
  • What should they never do? (→ constraints)

This module teaches you each of these components — and how they combine into a system that produces reliable, high-quality output across your PM workflow.

Concept 02

The Anatomy of a Great System Prompt

A system prompt isn't a conversation opener. It's a contract between you and your AI — specifying exactly what it does, how it does it, and what success looks like.

The single highest-leverage thing you can do to improve AI output quality is write a better system prompt. Not a better question — a better job description. The system prompt persists across every conversation in a Claude Project. It defines the specialist you're building.

The Contract-Style Pattern

The most effective system prompts follow a five-section pattern. Think of it as a contract: clear terms, no ambiguity, with examples of what compliance looks like.

System Prompt — Weekly Stakeholder Update Skill
<role> You are a PM communications specialist. You transform raw project data into concise stakeholder updates for senior leadership. </role> <success_criteria> - Lead with the single most important decision or blocker - Keep total length under 400 words - Include exactly 3 sections: Progress, Risks, Asks - Every risk must have a mitigation plan or a specific ask - Use concrete numbers, not vague qualifiers ("3 of 7 stories complete" not "good progress") </success_criteria> <constraints> - Never use jargon the VP of Marketing wouldn't understand - Never invent metrics or dates — if data is missing, flag it explicitly - Never bury bad news below the fold </constraints> <uncertainty_handling> If you're missing information needed for a complete update, state what's missing and suggest where to find it rather than guessing. </uncertainty_handling> <output_format> ## [Project Name] — Week of [Date] **Top Line:** [One sentence: the most important thing leadership needs to know] ### Progress [2-3 bullet points with concrete metrics] ### Risks [Each risk with severity and mitigation/ask] ### Asks [Numbered list of specific decisions or resources needed, with deadlines] </output_format>

Notice what this prompt doesn't do: it doesn't describe how LLMs work, it doesn't beg the model to try its best, it doesn't use threatening language. It specifies the job clearly — role, success criteria, constraints, how to handle uncertainty, and the exact output format. Claude fills in the rest.

Why This Works: The Job Description Analogy

When you hire someone, you don't write them a pep talk. You write a job description: here's what you do, here's what success looks like, here's what you never do. A system prompt works the same way. The five sections map directly:

SectionJob Description EquivalentWhat It Does
RoleJob title + teamFocuses the model's behavior and vocabulary
Success criteriaKPIs / OKRsDefines what "good" looks like — measurably
ConstraintsCompliance / guardrailsDefines what to never do — prevents common failures
Uncertainty handling"Escalation policy"Tells the model what to do when it's not sure — reduces hallucination
Output formatTemplate / deliverable specEliminates format guesswork — output is consistent every time

The XML Structure Advantage

You'll notice the example uses XML tags like <role> and <constraints>. This isn't decorative. XML tags are the most effective structuring method for Claude specifically. They help Claude parse complex prompts unambiguously, especially when you're mixing instructions, context, examples, and variable inputs. Multiple independent tests have confirmed XML tags produce more reliable instruction-following than Markdown headers or numbered lists.

Practical Tip

Use consistent, descriptive tag names across your prompts. <role>, <instructions>, <context>, <examples>, <output_format> — pick names that mean something and reuse them. Nest tags when content has a natural hierarchy. Claude will parse them reliably and you'll build a personal prompt vocabulary that works across all your skills.

The Length Sweet Spot

The best system prompts are 150–300 words. That's a half-page, not a novel. Structure beats length every time. A 200-word prompt with clear XML sections, specific success criteria, and one good example will outperform a 1,500-word wall of text. If your prompt is getting long, you probably need to split it into a system prompt + uploaded reference document instead.

Concept 03

What Claude 4.x Actually Needs From You

Claude's latest models follow instructions literally. This changes how you write prompts — in ways that are better for PMs once you understand the shift.

If you used earlier Claude models (3.5 or before), you might have gotten away with vague instructions. The model would infer what you probably meant and add extras you didn't ask for. Claude 4.x — including Opus 4.6 and Sonnet 4.6 — behaves differently: if you don't ask for it, you won't get it.

This sounds like a limitation. It's actually an upgrade. You get predictable, controllable output. No more unwanted additions, no more inconsistent formatting between sessions. But it requires you to be explicit about what you want.

The Five Behavioral Shifts

BehaviorWhat ChangedWhat to Do Now
Literal following Model no longer infers "above and beyond" behavior from vague prompts Be explicit: "Include 5 competitive alternatives with pricing" not "be comprehensive"
Aggressive language hurts "CRITICAL!", "YOU MUST", "NEVER EVER" overtrigger the model and produce worse results Use calm, direct instructions: "Use this tool when..." not "ALWAYS USE THIS TOOL"
Motivated constraints work best Explaining why a rule exists helps the model generalize it to edge cases "Format as prose paragraphs because this will be read aloud at standup" beats "Do not use bullet points"
More concise by default Model may skip summaries, jump directly to action, provide shorter responses If you want detail, ask: "Provide a thorough analysis with supporting evidence for each point"
Proactive tool use Model is more aggressive about using tools; previous "nudge" language now causes overtriggering Remove old prompts like "If in doubt, use [tool]" — model will trigger appropriately on its own

The Golden Rule Test

Anthropic's official prompting guide offers this simple test: show your prompt to a colleague with minimal context on the task and ask them to follow it. If they'd be confused, Claude will be too. If your colleague would ask "what do you mean by 'comprehensive'?" or "how long should this be?" or "who is the audience?" — your prompt is missing critical information.

Vague — Claude Guesses
"Write a competitive analysis for my product."
Missing: Which competitors? What dimensions? What format? For whom? How long? Claude will produce something — but it will be generic and probably wrong for your context.
Explicit — Claude Delivers
"Compare Notion, Linear, and Asana on these dimensions: pricing, API capabilities, and enterprise security. Format as a comparison table with our product (Acme PM) in the fourth column. Audience: engineering leadership evaluating migration. Max 500 words plus table."
Clear competitors, clear dimensions, clear format, clear audience, clear length. Claude can hit this spec on the first attempt.
Counterintuitive

"Don't" instructions are sometimes more effective than "do" instructions — but only when motivated. "Never mention competitor pricing as an advantage because our legal team flagged this as a compliance risk" works better than both "Don't mention pricing" (unmotivated negative) and "Focus on feature differentiation" (vague positive). The reason matters because it helps Claude apply the constraint intelligently to cases you didn't anticipate.

Concept 04

Claude Projects as Your Skill Management System

Each Claude Project is a specialist on your team. System prompt = skill definition. Uploaded docs = reference library. You're building a personal AI team, one Project at a time.

A Claude Project is the simplest way to turn what you learned in Concepts 01–03 into something permanent. Instead of rewriting your system prompt every session, you create a Project with instructions that persist across every conversation inside it. Upload relevant documents and they become the Project's always-available reference library.

Think of your Projects as specialists sitting at desks, each with a clearly defined job:

ProjectSystem Prompt FocusUploaded DocsYou Use It For
Sprint Reporter Transforms raw sprint data into leadership-ready updates Stakeholder update template, team roster, OKR doc Weekly status emails, sprint review prep
Feedback Synthesizer Extracts patterns from customer feedback, categorizes by theme and severity Product taxonomy, persona definitions, last quarter's themes Support ticket analysis, NPS deep-dives, interview synthesis
Competitive Analyst Compares features, pricing, positioning against specific competitors Competitive landscape doc, your product's feature matrix, pricing page Win/loss analysis, feature comparison tables, market positioning

The Three Context Decisions

For every Project, you make three decisions about where information lives:

  • System prompt: Instructions that apply to every conversation. Role, success criteria, constraints, output format. Changes rarely. This is the "job description."
  • Uploaded documents: Reference material the specialist needs on their desk. Product docs, templates, style guides, competitor data. Changes occasionally — update when the source material changes.
  • Per-conversation context: The specific data for this task — the sprint's JIRA export, this week's support tickets, the specific competitor to analyze. Changes every session. This is what you paste or connect via MCP.
The Key Design Principle

Stable information goes in the system prompt and uploaded docs. Variable information goes in the conversation. If you find yourself repeatedly pasting the same context, it probably belongs in the Project's uploaded docs. If your system prompt changes with every task, you're conflating instructions with data — split them.

Building Your First Three Projects

Your Module 02 deliverable identified three highest-leverage PM tasks. Those are your first three Projects. For each one:

  1. Write a contract-style system prompt using the five-section pattern from Concept 02. Keep it under 300 words.
  2. Upload 2–3 reference documents that the specialist would need. Your product's feature matrix, the stakeholder template you want output to match, the personas your team uses.
  3. Run the task three times with different inputs. Is the output consistently good? If not, the issue is almost always in the system prompt — unclear success criteria or missing constraints.
Module 02 Connection

Your PM work audit identified tasks with specific inputs, outputs, audiences, and frequencies. That specificity is exactly what makes a good Project. "Weekly synthesis of Zendesk tickets into themed report for eng standup" becomes a Project with a system prompt defining the synthesis methodology, uploaded docs containing the team's product taxonomy, and per-conversation input of this week's ticket export.

Concept 05

MCP — Your AI's Integration Layer

The Model Context Protocol is the open standard that lets Claude talk to your work tools. Think USB-C for AI: one standard, works everywhere.

You've built Projects with system prompts and uploaded docs. But there's a problem: the data your specialist needs is often live. This week's JIRA tickets are different from last week's. Today's Slack highlights aren't yesterday's. You've been copy-pasting this data manually — which defeats half the point of having an AI system.

MCP (Model Context Protocol) fixes this. It's an open standard — originally created by Anthropic in late 2024, now governed by the Linux Foundation and adopted by Claude, ChatGPT, VS Code, Gemini, and hundreds of other tools. When you enable an MCP connector, Claude can pull live data from your work tools directly, without copy-paste.

How It Works in Practice

Setup takes seconds for most connectors: go to Claude's settings, find the connector, click "Connect," authorize via OAuth. Once enabled, Claude can use the connector in any conversation where you've toggled it on.

The practical advice from teams already using MCP: don't connect everything at once. Start with the two or three tools you use most. Build workflows that save time with those. Then expand. The directory grows quarterly — Anthropic has confirmed new integrations regularly ship.

Security Note

MCP connectors use delegated permissions — Claude can only access what you can access. This means if you can see a confidential Slack channel, so can Claude when the connector is active. Be deliberate about which connectors you enable for which conversations. Review permissions during setup. Community MCP servers carry additional risk — stick with official connectors from the directory for sensitive workflows, and limit write access until you trust the workflow.

Concept 06

Few-Shot Examples and Output Anchoring

One well-chosen example of ideal output is worth more than ten paragraphs of instructions. This is the single most reliable way to control quality and consistency.

You've written a contract-style system prompt with clear success criteria. Your Projects are set up. Output is better — but maybe not consistent. Sometimes the format drifts. Sometimes the tone shifts. Sometimes the depth varies between sessions.

Examples fix this. When you show Claude what great output looks like for your specific task, consistency jumps dramatically. This technique — called "few-shot prompting" — is one of the most reliable tools in your workspace design kit.

The Sweet Spot: 2–3 Examples

Anthropic's official guidance: include 3–5 examples for best results. In practice, for PM tasks, 2–3 carefully chosen examples hit the sweet spot. Here's why:

  • 1 example (one-shot) often suffices for format and tone. Start here.
  • 2–3 examples cover format + edge cases. This is the reliable range for most PM skills.
  • 5+ examples can backfire — the model starts mimicking surface patterns (word choice, sentence structure) instead of understanding the underlying task. You get output that looks like a copy of your examples rather than a thoughtful application of the pattern.

The Golden Examples Technique

The best examples aren't hypothetical — they're your own best work. Every time Claude produces output you're particularly happy with, save it. These "golden examples" become the anchors for your system prompt:

System Prompt Excerpt — Example Section
<examples> <example> <!-- A real output you were happy with, anonymized if needed --> ## ProjectX — Week of March 10 **Top Line:** API migration is 2 days behind schedule due to unexpected auth compatibility issues; need eng director decision on scope cut by Friday. ### Progress - 5 of 8 endpoints migrated and passing integration tests - Performance benchmarks show 40% latency improvement vs legacy - Partner notification emails sent to 12 of 15 affected accounts ### Risks - **Auth compatibility (High):** OAuth token refresh breaks for 3 enterprise clients using legacy SDK. Mitigation: Engineering proposed a shim layer, needs 2 additional days. Ask below. ### Asks 1. Approve 2-day extension for auth shim (eng director, by Fri March 14) 2. Confirm partner comms go out Monday regardless of auth fix (PM lead, by Fri) </example> </examples>

When Claude sees this example alongside your system prompt, it understands: this is the exact format, this is the level of specificity for metrics, this is how risks get structured, this is what an "ask" looks like. No amount of verbal instruction communicates this as clearly as one concrete example.

Make Examples Diverse

If you use 2–3 examples, make them cover different situations — not just variations of the same scenario. For a stakeholder update skill: one example with good news (on-track sprint), one with a blocker (delayed timeline with an ask), and one with mixed signals (some features ahead, others behind). This teaches Claude the pattern, not a template.

Concept 07

Designing MCP-Connected Workflows

The real power isn't connecting tools. It's combining your system prompt, your examples, and your live data into a workflow that runs in minutes instead of hours.

You've got Projects with great system prompts and examples. You've got MCP connectors pulling live data. Now you combine them — and this is where things get transformative.

Consider the weekly sprint summary that most PMs spend 45–90 minutes on every Friday:

Without MCP · 45–90 min
Manual Sprint Summary
  1. Open JIRA, export sprint board as CSV
  2. Open Slack, scroll through #engineering channel
  3. Open Google Docs, copy stakeholder template
  4. Manually compile data into template
  5. Paste into Claude Project, ask for formatting
  6. Review, edit, send
With MCP · 10–15 min
Connected Sprint Summary
  1. Open your Sprint Reporter Project (MCP enabled)
  2. Type: "Pull this sprint's data and highlights"
  3. Claude queries JIRA for sprint status, reads Slack #engineering for key threads
  4. Claude generates stakeholder update using your system prompt, examples, and live data
  5. Review, edit, send

The time savings are real — but the quality shift matters more. When Claude can see the actual JIRA data instead of your hastily pasted export, it catches things you might miss: stories that were moved to the backlog mid-sprint, a blocker that was flagged on Tuesday but resolved by Thursday, velocity trends that only appear when comparing against the previous sprint.

Design Principles for Connected Workflows

  • Start with your highest-leverage Project. Don't MCP-connect everything at once. Pick the one Project from your Module 02 targets that uses the most external data. Connect that one first. Get the workflow right. Then expand.
  • Separate data-pull from analysis. A useful pattern: first ask Claude to pull and summarize the raw data ("Show me this sprint's JIRA status and Slack highlights"), review what it found, then ask for the analysis ("Now generate the stakeholder update"). This two-step approach lets you verify the data before the synthesis.
  • Tell Claude which connectors to use. If multiple connectors are enabled, be specific: "Search Slack #product-launches for any threads about the API migration" rather than "find what people said about the migration." Specificity reduces noise.
Counterintuitive

The biggest productivity gains from MCP aren't in tasks you already do — they're in tasks you used to skip. Most PMs don't do comprehensive competitive monitoring weekly because it takes too long. With a Competitive Analyst Project connected to web search, you can run it in 15 minutes. Most PMs don't synthesize all support tickets before a planning meeting. With a Feedback Synthesizer connected to your support tool, you can. MCP makes the "would be nice if I had time" category suddenly feasible.

Concept 08

Prompt Chaining vs. Mega-Prompts

Complex tasks fail in one shot. Break them into steps. Each step is simpler, more reliable, and — critically — easier to evaluate and improve independently.

It's tempting to write one massive prompt that does everything: "Research competitors, synthesize findings, compare on 12 dimensions, create a table, draft an executive summary, and suggest three strategic responses." That's a mega-prompt. It will produce output, but it will be unreliable — some sections great, others mediocre, and you won't know which part of the prompt caused which problem.

Prompt chaining breaks complex tasks into sequential steps. Each step has one job. The output of step one becomes the input for step two.

1
Research
"Search for recent news and updates about Notion, Linear, and Monday.com. Focus on enterprise features, pricing changes, and new AI capabilities launched in the last 6 months."
↓
2
Synthesize
"Using the research above, create a structured comparison on these dimensions: pricing tiers, API capabilities, AI features, enterprise security certifications, and integration ecosystem."
↓
3
Draft
"Based on the comparison, draft a 300-word executive summary for our VP of Product. Lead with the most significant competitive threat. End with 2 recommended strategic responses."
↓
4
Review
"Review the executive summary against these criteria: (1) every claim has supporting data, (2) no jargon the VP wouldn't know, (3) recommendations are specific and actionable. Flag any issues."

Each step is simple, testable, and improvable independently. If the research step is weak, you fix that prompt without touching the synthesis logic. If the executive summary is too vague, you tighten step 3 without re-running the research. This separation is what makes chaining powerful — and it's exactly what Module 04 will help you evaluate systematically.

When Chaining vs. When a Single Prompt

Claude 4.x models have strong adaptive thinking — they often handle multi-step reasoning internally without you breaking it into pieces. Explicit chaining is still valuable when:

  • You need to inspect intermediate outputs. "Show me the raw data before you analyze it."
  • Different steps have different quality issues. Research might be great but synthesis weak — you need to know which step to fix.
  • The task spans multiple conversations or sessions. Each step's output can be saved and resumed later.
  • You want a self-correction loop. Generate → review against criteria → refine. This "generate then critique" pattern is the most common production chaining pattern.
Module 04 Preview

Prompt chaining becomes essential when you start evaluating your skills. In Module 04, you'll run 15+ inputs through each skill and label outputs as pass or fail. If a skill uses a single mega-prompt, you can only fix the whole thing at once. If it uses a chain, you can identify exactly which step produces the failures and fix just that step. Build chains now — you'll thank yourself in Module 04.

Concept 09

Claude Skills — From Projects to Portable Expertise

Projects are great for personal use. Skills are the production version: portable, versioned, shareable, and executable across every Claude surface.

You've been building Projects — each one a specialist workspace with a system prompt and uploaded docs. Skills take this further. A Skill is a packaged folder of instructions that Claude loads automatically when relevant. The analogy that captures the difference: if MCP is the kitchen (access to tools and ingredients), Skills are the recipes (how to use them).

Projects vs. Skills

Project
Subject-focused workspace
Contains everything about one topic: Q3 planning docs, competitive landscape, customer research. You manually open the right Project and work inside it. Good for personal, conversational use — where you're the one choosing which specialist to engage.
Skill
Behavior-focused instructions
Defines how Claude handles a specific type of task: generating stakeholder updates, analyzing feedback, creating competitive tables. Loads automatically when Claude detects a matching request. Portable across Claude.ai, Claude Code, and API.

How Skills Work: Progressive Disclosure

Skills use a three-level architecture that keeps Claude efficient. Only the minimum information loads at each stage:

L1
Metadata
Always loaded at startup
The YAML frontmatter: skill name and description. Just enough for Claude to know when this skill is relevant. Like a job title on a team roster.
~100 tokens per skill
L2
Instructions
Loaded when skill is triggered
The SKILL.md body: full workflow, examples, constraints. Like opening the specialist's playbook. Only loads when Claude thinks this skill matches your request.
<5,000 tokens typically
L3
Resources
Loaded as needed
Additional files: scripts, templates, reference docs. Like the specialist pulling a reference manual off their shelf. Only accessed when specifically needed.
Effectively unlimited

This architecture means you can install many Skills without flooding Claude's context. Claude reads only the metadata for each installed Skill. The full instructions only load when a specific skill triggers — keeping the context window lean.

When to Graduate from Project to Skill

Not everything should be a Skill. Use this decision framework:

  • Stay with Projects when you need subject-specific knowledge (uploaded docs, project context) and the skill is just for your personal use.
  • Graduate to a Skill when you've refined a workflow through multiple iterations (it works reliably), you want it to trigger automatically across any conversation, or you want to share it with your team.
  • Skill + MCP together for the most powerful patterns. The Skill tells Claude the recipe; the MCP connector provides the live ingredients. Example: a "Sprint Summary Skill" that knows your update format, combined with a JIRA connector that provides the live data.
Skills Are an Open Standard

Anthropic published Agent Skills as an open standard. The same SKILL.md format works across Claude.ai, Claude Code, Cursor, Gemini CLI, and other compatible tools. You're not locked in. Skills are distributed as folders — upload to Claude.ai via Settings → Capabilities → Skills, or place directly in a Claude Code skills directory. Organization admins can deploy skills workspace-wide.

Concept 10

MCP Apps — Your Conversation Becomes a Workspace

MCP connectors pull data into your conversation. MCP Apps render interactive UIs inside it. You filter dashboards, edit timelines, and review designs — without ever leaving Claude.

MCP connectors (Concept 05) let Claude read data from your tools. MCP Apps go further: they render interactive interfaces inside your conversation. Instead of Claude describing your Amplitude dashboard in text, the dashboard appears in the chat — and you can filter, drill down, and explore it.

What MCP Apps Look Like in Practice

When an MCP App is active, your conversation window becomes a hybrid workspace:

  • Amplitude: Charts and dashboards render inline. You can ask Claude to filter by segment, change date ranges, and compare cohorts — all within the chat.
  • Asana / Linear: Project timelines and task boards appear as interactive components. You can edit due dates, reassign tasks, and create new items without switching tabs.
  • Figma: Design files render for review. You can annotate, comment, and reference specific components in your conversation with Claude.
  • Slack: Message composers render inline. Draft messages, preview them in context, edit, and send — all from within Claude.

For PMs, this collapses the "tab-switching tax." Instead of operating across six different tools — each with its own interface, its own login, its own mental model — you operate from one environment. Claude becomes your workspace, not just your assistant.

Connectors vs. Apps

Connectors are for data access. Apps are for interactive work. Use a connector when you need Claude to pull information and analyze it (searching Slack, reading JIRA data, querying a database). Use an MCP App when you want to interact with the tool visually — filtering dashboards, editing project plans, reviewing designs. Many tools offer both, and they complement each other.

Concept 11

Structured Outputs for Reliable Data Extraction

Stop hoping Claude returns the right format. Structured outputs enforce a JSON schema — guaranteed conformance, every time.

You learned about structured outputs in Module 01 as a concept. Now you put them to work. When you need Claude to extract data into a specific shape — a competitive feature table, categorized support tickets, a formatted spec — structured outputs enforce the schema. The output isn't "pretty close" to JSON. It's validated against your exact specification.

When to Use Structured Outputs

Structured outputs shine when the downstream consumer of Claude's work is a system, not a human. Common PM use cases:

  • Competitive feature matrices: Extract features, pricing, and capabilities into a consistent table format you can import into a spreadsheet or database.
  • Support ticket categorization: Classify each ticket by theme, severity, feature area, and sentiment — structured so you can aggregate and chart the results.
  • Meeting action items: Extract action items with owner, deadline, and status fields from meeting notes or transcripts.
  • User story extraction: Pull structured user stories (role, action, benefit, acceptance criteria) from freeform requirements documents.

How Structured Outputs Work with MCP

The combination is powerful: MCP pulls live data in, structured outputs shape data going out. A workflow might look like:

  1. MCP connector pulls the last 200 support tickets from your help desk
  2. System prompt defines your categorization taxonomy (product area, severity, theme)
  3. Structured output ensures each ticket is categorized into exactly the fields your analytics dashboard expects
  4. Result: A clean, importable dataset — no manual cleanup needed
Module 01 Connection

In Module 01, you learned that structured outputs guarantee JSON schema conformance — not "please try to return JSON" but validated output against a spec you define. Here's the practical payoff: every skill you build that produces data (not prose) should use structured outputs. The format consistency means your outputs are reliably importable into spreadsheets, dashboards, and other tools without manual cleanup every time.

Concept 12

Thinking Like an MCP Product Manager

"Should your product have an MCP server?" is the 2026 version of "Should we have an API?" If you're a PM at a SaaS company, this question is already on your roadmap — or it should be.

You've been using MCP as a consumer — enabling connectors to pull data into your AI workflows. Now think about it as a product strategy: what happens when your own product exposes an MCP server?

The logic mirrors the API economy of the 2010s. When every SaaS product built a REST API, it wasn't because APIs were trendy — it was because customers demanded integration. MCP is following the same arc. Forrester predicts 30% of enterprise app vendors will launch MCP servers in 2026. The question isn't whether to build one, but when and what to expose.

What an MCP Server Exposes

MCP servers publish three types of capabilities to AI clients:

⚡
Tools
Actions the AI can perform. "Create a ticket," "update a status," "search projects." Equivalent to API endpoints, but discoverable by any MCP client.
📄
Resources
Data the AI can read. "Project details," "user profiles," "ticket history." Read-only data sources that provide context for AI reasoning.
💬
Prompts
Reusable templates for common workflows. "Summarize this project," "create a weekly report." Pre-built interaction patterns for your product.

The N×M Problem MCP Solves

Before MCP, if your product wanted to work with 5 AI platforms (Claude, ChatGPT, Gemini, Copilot, Cursor), you needed 5 custom integrations — each with its own auth flow, data format, and maintenance burden. With MCP, you build one server and it works with all of them. The 97 million monthly SDK downloads confirm: the standard has won.

The PM Decisions

If you're speccing an MCP server for your product, these are the decisions that matter:

  • Read-only first, write later. Start by exposing resources (data that AI can read). This is low-risk and immediately valuable — customers can query your product's data through their AI workflows. Add write tools (create, update, delete) once you've validated the use cases and security model.
  • Scope permissions deliberately. MCP uses delegated permissions — the AI can only do what the authenticated user can do. But that's still a broad surface. Consider tool-level access controls: expose search and read tools to all users, but restrict create and delete tools to admins.
  • Think in workflows, not endpoints. Don't just mirror your REST API as MCP tools. Design for how AI agents actually use your product. A "get-project-summary" tool that returns a synthesized view is more useful to an AI than separate calls to list-tasks, list-milestones, get-project-details that the AI has to orchestrate itself.
  • Prompt injection matters. Content from your product flows into AI context windows. If a malicious user can inject text into a field your MCP server returns, that text could influence AI behavior. Sanitize outputs. This is the MCP equivalent of XSS prevention.
The Strategic View

MCP servers are becoming the new competitive moat for SaaS products. When a customer's AI assistant can seamlessly pull data from your tool, create items, and trigger workflows — all in natural language — the switching cost increases dramatically. The product that's deeply embedded in a customer's AI workflow is harder to replace than the product that just has a good UI. This is the "API moat" thesis, updated for the AI era. Module 03 Project

Build Your AI Toolkit: 3 Projects + MCP Workflow + Product Spec

This is the most hands-on project in the curriculum. You're not working with hypothetical tasks — you're building the actual AI system for the three highest-leverage PM tasks you identified in Module 02. Everything you build here becomes the subject of your first evaluation in Module 04.

  1. Build your first Claude Project. Take your #1 target from the Module 02 audit. Write a contract-style system prompt (role, success criteria, constraints, uncertainty handling, output format). Upload 2–3 reference documents. Run the task three times with different inputs and iterate the system prompt until output is consistently good.
  2. Build Projects 2 and 3. Repeat for your other two audit targets. For at least one, include 2–3 few-shot examples in the system prompt using the golden examples technique. Compare output consistency with and without examples.
  3. Connect MCP to your highest-data Project. Pick the Project that relies most on live data (sprint summaries, feedback synthesis, competitive analysis). Enable the relevant MCP connector(s). Run the workflow with MCP connected vs. with manually pasted data. Document what changed: time saved, data completeness, quality differences.
  4. Write an MCP product spec. If your product had an MCP server, what would it expose? Write a brief spec: 3–5 tools (actions AI can take), 3–5 resources (data AI can read), and 1–2 prompt templates (common workflows). Include notes on permission scoping and security considerations. One page is enough.
  5. Document your toolkit. Compile a simple reference: each Project's name, purpose, system prompt summary, connected tools, and one example of output. This becomes your personal AI toolkit reference — and the input for Module 04's evaluation.
Deliverable → 3 working Claude Projects with system prompts, 1 MCP-connected workflow with before/after comparison, 1 MCP product spec, and a toolkit reference doc. Your best-performing skill becomes the subject of Module 04's evaluation.
